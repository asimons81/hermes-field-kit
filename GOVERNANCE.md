# Governance

## Model

Hermes Field Kit is maintainer-led. Tony Simons (`@asimons81`) is the initial maintainer and final decision-maker for scope, quality, releases, and moderation.

This model is intentionally lightweight during the foundation and early-release phases.

## Decision principles

Project decisions prioritize:

1. User safety and privacy
2. Reproducible real-world usefulness
3. Compatibility with Hermes Agent
4. Reviewability and maintenance cost
5. Repository coherence
6. Quantity only after quality

## Contributions

Anyone may propose issues or pull requests. Maintainers may accept, request changes, defer, or decline contributions. Declining a contribution does not imply that the work lacks value; it may fall outside the project's deliberately narrow scope.

## Material changes

Changes to the admission rule, skill specification, catalog format, licensing, security model, or release policy require:

- A public issue explaining the problem
- A documented decision in the pull request
- A migration plan when existing consumers would be affected

## Maintainers

Additional maintainers may be invited after sustained, high-quality contributions and demonstrated judgment around privacy, safety, and review standards.

Maintainers are expected to disclose conflicts of interest and recuse themselves when necessary.

## Releases

The maintainer controls versioning, tags, and release notes. A pull request must not create a release or tag unless the release is explicitly in scope.

## Changes to governance

Governance changes follow the material-change process above and are recorded in `CHANGELOG.md`.
